## 1. Пользователи и роли

- Создаёт только админ. Самостоятельной регистрации нет.
- Три роли: `admin`, `teacher`, `student`
- Админ может деактивировать любую учётку (soft delete / флаг active)

**users** — логин, пароль, роль

**teachers** — ФИО, user_id → users.id

**students** — ФИО ребёнка, дата рождения, ФИО родителя, телефон, откуда узнали, user_id → users.id

> Админ: только логин/пароль в users, отдельной таблицы нет.

## 4. Группы

**groups** — название, возрастной диапазон, branch_id → branches.id, teacher_id → teachers.id

- Один основной педагог на группу, назначается админом
- Педагог может вести несколько групп
- Ученик может быть в нескольких группах — связь `student_groups(student_id, group_id)`
- Замены педагога на конкретном занятии — см. слайс 5

## 5. Расписание

### Модель

**Что заводит админ:**
Один раз в сентябре создаёт шаблоны. Затем генерирует конкретные занятия на учебный год (вручную, кнопкой "сгенерировать").

**schedule_templates** — день недели, время начала, группа, педагог, зал

**classes** — дата, время начала, группа, педагог, зал, template_id → schedule_templates.id

**Как отображается:**
Приложение читает готовые строки из `classes` за нужный период. Никаких вычислений на лету.

**Замена педагога на конкретное занятие:**
Админ открывает запись в `classes` и меняет педагога. Шаблон не трогается — все остальные занятия остаются без изменений.

## 6. Посещаемость

**attendance** — class_id → classes.id, student_id → students.id, present (boolean), комментарий (опционально)

- По умолчанию все ученики группы присутствуют. Педагог/админ отмечают только отсутствующих.
- Отмечать может педагог и админ.

---

## 7. Прайс-лист

**price_list_items** — название, цена, дата начала, дата окончания (опционально)

- Создаёт только админ
- Позиции без даты окончания висят всегда (разовое занятие, носки, форма и т.д.)
- Позиции с датой окончания — временные (взнос на соревнования, мастер-класс и т.д.)
- Примеры: разовое занятие, взнос на соревнования, тренерский сбор, проезд/отель, мастер-класс, форма, мерч, носки

---

## 8. Начисления

**charges** — student_id → students.id, price_list_item_id → price_list_items.id, сумма, статус (оплачено / не оплачено), дата создания

- Создаёт только админ — на конкретного ученика или на всю группу (система создаёт запись каждому ученику группы)
- Сумма берётся из прайса, но можно редактировать вручную
- При создании указывается статус: оплачено сразу (покупка на месте) или нет (долг)

---

## 9. Оплаты

**payments** — сумма, дата, student_id → students.id, принял (user_id → users.id)

**payment_charges** — payment_id → payments.id, charge_id → charges.id (один платёж может закрывать несколько начислений)

### Абонемент — открытый вопрос (OQ-1)

Две рассмотренных модели:

**А) Платёж с периодом** — абонемент это товар из прайса, при оплате указывается месяц (март 2026). Можно платить за несколько месяцев вперёд. Минус: система не знает кто должен заплатить за текущий месяц.

**Б) Начисление на группу** — каждый месяц админ массово создаёт начисление "абонемент март" на всех учеников группы. Система знает должников. Минус: нельзя заплатить вперёд.

> Решение зависит от ответа Димы на OQ-1.
