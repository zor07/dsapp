# Описание функционала системы — MVP
### Танцевальная команда КИД

---

## 1. Назначение системы

Веб-приложение для управления танцевальной школой. Заменяет бумажную книгу учёта, журнал посещаемости и разрозненные заметки администратора — переводя всё это в единую цифровую среду.

Главная задача первой версии: дать администратору школы инструмент для учёта платежей с множеством позиций и автоматического контроля долгов, а педагогам — возможность отмечать посещаемость с телефона.

Приложение открывается в браузере (телефон или компьютер), скачивать ничего не нужно.

---

## 2. Роли пользователей

| Роль | Кто это | Что умеет |
|------|---------|-----------|
| **Администратор** | Кристина и другие назначенные сотрудники | Полный доступ: управляет учениками, группами, платежами, расписанием, педагогами |
| **Педагог** | Преподаватели школы (до 5 человек) | Отмечает посещаемость на своих занятиях, видит список своей группы, управляет своими окнами в расписании |
| **Ученик / Родитель** | Один аккаунт на клиента (ребёнок = клиент) | Видит своё расписание, начисления и историю оплат |

Аккаунты для учеников и педагогов создаёт только администратор.

---

## 3. Границы MVP

### Входит в эту версию

- Картотека учеников и групп
- Многопозиционный учёт платежей (начисления, оплата, задолженности)
- Перерасчёт абонемента (новый ученик / болезнь со справкой)
- Журнал посещаемости
- Расписание групп (фиксированное на сезон)
- Окна педагогов (отметка свободного времени)
- Три роли с разграниченным доступом
- Полностью на русском языке
- Мобильная и десктопная версии

### Не входит в эту версию

| Функция | Когда планируется |
|---------|-------------------|
| Модуль соревнований и поездок | Следующая итерация |
| Индивидуальные занятия (запись, учёт) | Следующая итерация |
| Онлайн-оплата | Будущее |
| Уведомления (напоминания об оплате) | Будущее — код подготовлен, доставка не реализована |
| Публичная регистрация учеников | Будущее |
| Статистика и аналитика | Будущее |

## 4. Функциональные блоки

---

### 4.1 Управление учениками и группами

**Что это:** Единая база данных учеников школы с привязкой к группам. Заменяет разрозненные контакты в телефоне и ручные записи.

**Кто использует:** Администратор — полный доступ. Педагог — просмотр своей группы.

**Ключевые сценарии:**

1. Администратор создаёт профиль нового ученика: ФИО ребёнка, дата рождения, ФИО родителя / ответственного, номер телефона, откуда узнали о школе.
2. Администратор создаёт логин и пароль для ученика и передаёт их родителю.
3. Администратор привязывает ученика к одной или нескольким группам.
4. Администратор создаёт новую группу: название, возрастной диапазон, филиал, основной педагог.
5. Администратор переводит ученика из одной группы в другую (например, при повышении уровня).
6. Администратор деактивирует профиль ученика, который ушёл из школы (данные сохраняются, доступ закрывается).
7. Педагог открывает список своей группы и видит имена, фото (если загружено) и контактный телефон.

---

### 4.2 Учёт платежей

**Что это:** Центральный модуль системы. Заменяет бумажную книгу учёта, заметки в телефоне и Telegram-переписку. Позволяет начислять любое количество позиций каждому ученику и отслеживать статус оплаты.

**Кто использует:** Администратор — создаёт начисления, отмечает получение денег, видит всех должников. Педагог — фиксирует факт получения наличных. Ученик — видит только свои начисления и историю.

**Типы платежей** (создаются и редактируются администратором):
абонемент, разовое занятие, взнос на соревнование, тренерский сбор, форма / мерч, костюм, мастер-класс, носки, прочее.

**Ключевые сценарии:**

1. Администратор начисляет абонемент за текущий месяц сразу всей группе — одним действием, а не по одному.
2. Администратор создаёт новую позицию оплаты для группы (например, «Мастер-класс 15 апреля — 3 000 руб») — она появляется в личных кабинетах всех учеников этой группы со статусом «не оплачено».
3. Педагог или администратор фиксирует получение наличных двумя способами:
   - Из профиля конкретного ученика: открыл профиль → выбрал позиции → отметил оплату.
   - Через отдельный экран «Принять оплату»: нашёл ученика по имени → выбрал позиции → отметил оплату.
4. Администратор редактирует сумму начисления для конкретного ученика (например, при пересчёте абонемента из-за прихода в середине месяца или болезни со справкой). Логику пересчёта администратор считает самостоятельно.
5. Администратор удаляет ошибочно созданное начисление (с подтверждением действия).
6. Администратор видит список всех должников по школе или по конкретной группе с суммой долга.
7. Ученик заходит в свой кабинет и видит, что у него не оплачено и на какую сумму, а также историю прошлых платежей.

---

### 4.3 Журнал посещаемости

**Что это:** Цифровой аналог бумажного журнала. Педагог отмечает присутствие прямо с телефона во время или после занятия.

**Кто использует:** Педагог — отмечает посещаемость на своих занятиях. Администратор — видит полную историю по любому ученику и любой группе.

**Ключевые сценарии:**

1. Педагог открывает сегодняшнее занятие своей группы и видит список учеников.
2. Педагог видит список группы — все считаются присутствующими по умолчанию. Педагог отмечает только отсутствующих.
3. Педагог опционально добавляет комментарий к отсутствию (например, «болезнь», «предупредил заранее»).
4. Администратор открывает профиль ученика и видит его посещаемость за любой период — для обоснования перерасчёта абонемента.
5. Администратор видит сводную посещаемость группы за месяц (кто сколько раз пришёл).
6. Любой педагог группы может зайти и отметить посещаемость — не только основной.

---

### 4.4 Расписание и окна педагогов

**Что это:** Фиксированное расписание занятий на учебный год. Видно только авторизованным пользователям (не публикуется открыто). Дополнено возможностью для педагогов отмечать свои свободные окна.

**Кто использует:** Администратор — создаёт и редактирует расписание, назначает замены. Педагог — видит своё расписание, отмечает свободные окна. Ученик — видит расписание своей группы.

**Ключевые сценарии:**

1. Администратор создаёт расписание группы: день недели, начало и конец занятия, зал, педагог. Это делается один раз в сентябре.
2. Администратор назначает замену педагога на конкретное занятие (например, Воля заменяет Катю в эту пятницу).
3. Ученик открывает своё расписание и видит дни и время занятий своей группы.
4. Педагог открывает расписание всей школы с возможностью отфильтровать «только моё».
5. Администратор видит сводную загрузку залов по дням.

## 5. Экраны и навигация

### Администратор

| Экран | Что на нём |
|-------|-----------|
| Дашборд | Число должников, сумма долгов по школе, ближайшие занятия сегодня, последние полученные платежи |
| Список учеников | Все ученики с фильтром по группе, поиском по имени, статусом активности |
| Профиль ученика | Личные данные, группа, все начисления и статусы оплаты, история посещаемости |
| Список групп | Все группы с числом учеников и основным педагогом |
| Профиль группы | Ученики группы, расписание, педагог, общая посещаемость |
| Начисления | Создание новых позиций оплаты для группы или отдельного ученика |
| Учёт оплаты | Фиксация полученных наличных: ученик → сумма → позиции |
| Должники | Сводная таблица: кто, сколько должен, по каким позициям |
| Журнал посещаемости | По группе и дате: кто был, кто нет |
| Расписание | Сетка по дням и залам, все группы |
| Педагоги | Список педагогов, их группы, расписание |
| Создание аккаунта | Форма регистрации нового ученика или педагога |

### Педагог

| Экран | Что на нём |
|-------|-----------|
| Расписание | Полное расписание школы с фильтром «только моё» |
| Занятие → посещаемость | Список группы (все присутствуют по умолчанию), педагог отмечает отсутствующих и опционально — причину |

### Ученик / Родитель

| Экран | Что на нём |
|-------|-----------|
| Мои начисления | Список всех позиций: оплачено / не оплачено, суммы |
| История оплат | Хронология всех зафиксированных платежей |
| Моё расписание | Дни и время занятий группы |
| Мой профиль | Личные данные (только просмотр) |
